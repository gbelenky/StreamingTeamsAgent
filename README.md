# Streaming Teams Agent

A **deliberately simple** demo that shows how to stream answers from a **Microsoft Foundry** model into **Microsoft Teams** using the **Python Microsoft 365 Agents SDK**.

The bot is a "human history tutor" — you ask a question, the answer streams in token-by-token in the Teams chat (typing indicator → progressive text → final message with "Generated by AI" label and thumbs-up/down feedback).

| Component | What it is |
|---|---|
| Runtime | Python 3.12 managed by **uv** |
| Agent SDK | `microsoft-agents-hosting-aiohttp` 1.0 (GA) |
| Model | Microsoft **Foundry project** chat deployment (e.g. `gpt-4.1-mini`) |
| Host | Azure **App Service** (Linux, Python 3.12) |
| Channel | Azure **Bot Service** with Teams channel |
| Telemetry | **Application Insights** (Azure Monitor OpenTelemetry) |
| Auth to Foundry | **User-assigned Managed Identity** (no keys) |
| IaC | **Bicep**, deployed by **`azd`** |

No pre-release packages. No multi-agent orchestration. ~150 lines of agent code.

## 2. Clone → F5 (one-key debug in real Teams)

The repo is wired as a **Microsoft 365 Agents Toolkit** project. The Toolkit auto-creates the Entra app + Azure Bot registration + dev tunnel, sideloads the Teams app, and launches the Python agent — all on F5.

> 💡 **Why Teams (Edge), not the Playground?** The Playground is a quick functional tester, but it does not render token-by-token streaming visually — you only see the final message. Real Teams renders the typing indicator, progressive text, and the "Generated by AI" label correctly, which is the whole point of this demo.

### 2.1 First Run checklist (~3 min if you already have a Foundry project + M365 dev tenant)

Do these **in order**. Steps 3, 4 and 5 are easy to miss — without them, F5 fails before the bot ever sees a message.

1. **Clone and install deps**
    ```powershell
    git clone https://github.com/gbelenky/StreamingTeamsAgent c:\src\StreamingTeamsAgent
    cd c:\src\StreamingTeamsAgent
    uv sync
    ```
2. **Sign in to Azure** (so `DefaultAzureCredential` in `foundry_client.py` can call Foundry, and so Toolkit can create the Azure Bot resource)
    ```powershell
    az login
    ```
3. **Sign in to Microsoft 365** in the Toolkit sidebar
    - VS Code → **Microsoft 365 Agents Toolkit** sidebar icon → **Accounts** → sign in to a tenant that has **custom app upload enabled** (a [Microsoft 365 developer tenant](https://aka.ms/m365devprogram) works out of the box).
4. **Grant your account `Azure AI User` on the Foundry project**

    Azure Portal → your Foundry project → **Access control (IAM)** → **+ Add → Add role assignment** → role **Azure AI User** → assign to your user. (CLI alternative: `az role assignment create --assignee <your-upn> --role "Azure AI User" --scope <foundry-project-resource-id>`.)
5. **Fill in `env/.env.local.user`** (gitignored — create if it doesn't exist) with your Foundry endpoint:
    ```
    FOUNDRY_PROJECT_ENDPOINT=https://<your-proj>.services.ai.azure.com/api/projects/<your-proj>
    FOUNDRY_MODEL_DEPLOYMENT=gpt-4.1-mini
    ```
    Get the endpoint from the Foundry portal → your project → **Overview → Endpoint**. Use the deployment name of any chat model you've deployed in that project.
6. **F5 → "Debug in Teams (Edge)"** (the default profile in the Run/Debug dropdown)

Toolkit will then, in sequence: validate prerequisites → register an Entra app + secret → register an Azure Bot resource → start a dev tunnel on :3978 → write `BOT_ID` / `BOT_DOMAIN` / etc. into `env/.env.local` → start the Python agent → open Edge directly into the Teams chat with your bot. First run takes ~60s; subsequent F5s reuse the registrations and take ~10s.

Ask *"Tell me about the fall of Rome"* — you should see the typing indicator → progressive streaming text → final message with "Generated by AI" label and 👍/👎 buttons.

### 2.2 What the Toolkit set up for you

**Lifecycle YAMLs (one per Toolkit "environment"):**

- `m365agents.local.yml` — local F5 lifecycle for **Debug in Teams**: registers Entra app, registers Bot Framework bot, starts dev tunnel, sideloads Teams app, maps secrets into `.env`
- `m365agents.playground.yml` — kept in the repo for the optional "Debug in M365 Agents Playground" F5 profile (code-only iteration; no streaming UX)
- `m365agents.yml` — production lifecycle for `azd deploy` / Teams Admin Center publish; Azure resources are owned by Bicep under `infra/`

**Environment files:**

| File | Committed? | Written by | Read by | What's in it |
|---|---|---|---|---|
| `env/.env.local` | ✅ yes | Toolkit `provision` (Debug in Teams) | `m365agents.local.yml` | Non-secret IDs: `BOT_ID`, `TEAMS_APP_ID`, `BOT_DOMAIN`, `AZURE_SUBSCRIPTION_ID`… |
| `env/.env.local.user` | ❌ gitignored | **you** + Toolkit `provision` | `m365agents.local.yml` | Your `FOUNDRY_PROJECT_ENDPOINT`, `FOUNDRY_MODEL_DEPLOYMENT`; Toolkit-managed `SECRET_BOT_PASSWORD`, `CONNECTIONS__…CLIENTSECRET` |
| `env/.env.playground` / `.user` | ✅ / ❌ | optional | `m365agents.playground.yml` | Only used if you F5 with the Playground profile |
| `env/.env.dev` | ✅ yes | placeholder for cloud `provision` (unused if you deploy via `azd`) | `m365agents.yml` | Same shape as `.env.local` but for the `dev` Toolkit environment |
| `.env.sample` | ✅ yes | you (template) | `python -m streaming_teams_agent` (manual run) | Annotated template — copy to `.env` for §2.4 manual run |
| `.env` (project root) | ❌ gitignored | Toolkit Deploy step (merges committed + `.user` + injected) | the Python agent at runtime | The fully resolved set of env vars the agent actually sees |

**The golden rule:** never put secrets in committed `.env.*` files — put them in `.env.*.user` (which is gitignored). The Toolkit lifecycle merges both at Deploy time into the project-root `.env` that the Python process reads.

### 2.3 Troubleshooting F5

| Symptom | Fix |
|---|---|
| Teams F5 fails: `Invalid value for key "packageName" on TeamsManifestV1D17` | Manifest schema too old. Pull latest (`git pull`) — `appPackage/manifest.json` is on v1.19. |
| F5 hangs with debugpy `KeyboardInterrupt` mid-import of `azure.core` | Stale `justMyCode: false` in `.vscode/launch.json`. Pull latest (`git pull`) — current value is `true`. |
| Every model call returns 401 from Foundry | You skipped Step 4 in §2.1. Add **Azure AI User** role on the Foundry project to your account. |
| Every model call returns 404 `DeploymentNotFound` | `FOUNDRY_MODEL_DEPLOYMENT` doesn't match a real deployment. List deployments: Foundry portal → your project → **Models + endpoints** → use the **Name** column value. |
| Teams says "App can't be installed" / "Setup policies prevent custom app upload" | Your M365 tenant doesn't allow custom app upload. Use a [M365 dev tenant](https://aka.ms/m365devprogram) or ask your tenant admin to enable it in Teams Admin Center → Setup policies. |
| `devtunnel` step fails: not signed in | Run `devtunnel user login` once in a terminal, then re-F5. |
| **If you tried the Playground profile and got** `ERR_CONNECTION_REFUSED` on `http://localhost:56150` | Playground binary wasn't installed. Run once: `cd devTools\teamsapptester && npm install @microsoft/teams-app-test-tool --save`. |
| **If you tried the Playground profile and got** 401 on every message | Playground sends unauthenticated requests; pull latest — `m365agents.playground.yml` injects `CONNECTIONS__SERVICE_CONNECTION__SETTINGS__ANONYMOUS_ALLOWED=true`. |

## 3. Deploy to Azure with `azd`

```powershell
azd auth login
azd env new streaming-teams-demo

# Required env vars for the deployment
azd env set BOT_APP_ID                <bot-app-client-id>
azd env set BOT_APP_SECRET            <bot-app-client-secret>
azd env set BOT_TENANT_ID             <tenant-id>
azd env set FOUNDRY_PROJECT_ENDPOINT  https://<proj>.services.ai.azure.com/api/projects/<proj>
azd env set FOUNDRY_MODEL_DEPLOYMENT  gpt-4.1-mini

# Optional: grant the bot's managed identity access to your Foundry account
azd env set FOUNDRY_ACCOUNT_RESOURCE_ID /subscriptions/<sub>/resourceGroups/<rg>/providers/Microsoft.CognitiveServices/accounts/<account>

azd up
```

`azd up` provisions:
- User-assigned **Managed Identity**
- **Log Analytics** + **Application Insights**
- Linux **App Service Plan** (B1) + **App Service** (Python 3.12)
- **Azure Bot** with Teams channel enabled
- RBAC: **Azure AI User** on the Foundry account → MI

Then it builds and deploys the Python app via Oryx.

### Do I need a WebSocket-capable backend?

**No — WebSockets are not required.** Common misconception.

Teams "streaming" looks live, but on the wire it's plain HTTPS POSTs:

```
Teams ──POST /api/messages──► your bot   (one request, one response)
your bot ──POST activities──► https://smba.trafficmanager.net/...  (Bot Connector)
```

The SDK's `streaming_response.queue_text_chunk(...)` doesn't stream over a socket. It **batches chunks every ~1 s** and posts them as separate `typing` activities (with a `StreamInfo` entity) to the Bot Connector REST API. Teams renders them as a single growing message. `end_stream()` posts the final message with feedback buttons.

So the only network requirement for your host is:
- Accept inbound **HTTPS** on `/api/messages`
- Make outbound **HTTPS** to `smba.trafficmanager.net` and your Foundry endpoint

**Hosting options:**

| Host | Works? | Notes |
|---|---|---|
| **Azure App Service** (what this repo uses) | ✅ | Default. No special config. |
| **Azure Container Apps** | ✅ | Great alternative, same HTTPS model. |
| **Azure Functions** (HTTP trigger) | ⚠ works but awkward | Cold start hurts first-chunk latency; per-invocation timeout matters for long answers. |
| **AKS / VMs** | ✅ | Anywhere you can serve HTTPS. |
| **Static Web Apps / CDN-only** | ❌ | Needs a real backend. |

WebSockets would only matter if you used the **Bot Framework Streaming Extensions** transport (Direct Line). The **Teams channel does not use it** — REST over HTTPS is the only transport.

> 💡 On App Service, leave the **WebSockets** toggle off (you don't need it). Keep **Always On = true** (already set via Bicep) so the first message after idle doesn't pay a cold-start penalty that would make streaming feel sluggish.

## How the streaming actually works

In `agent.py`, every Teams message activity goes through:

```python
stream = context.streaming_response
stream.queue_informative_update("Consulting the history books…")  # typing indicator
for chunk in foundry_openai.chat.completions.create(..., stream=True):
    stream.queue_text_chunk(chunk.choices[0].delta.content or "")
await stream.end_stream()                                          # finalize
```

The SDK batches chunks (every ~1 s on Teams) and posts them as `typing` activities with `StreamInfo` entities — Teams renders them as live, progressive text. The final `end_stream()` posts the assembled message with "Generated by AI" + feedback buttons.
